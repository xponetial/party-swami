# Party Swami Hosting Checklist

This file tracks the changes, configuration, and verification work needed to move Party Swami from local development to a hosted production site.

Update this checklist whenever a new feature adds:
- a required environment variable
- a hosted-only service dependency
- a domain, callback, webhook, or email configuration
- a production verification step

## 1. Hosting Platform Setup

- Choose the production host for the Next.js app.
- Configure the production domain.
- Configure the preview/staging domain if we want a safe pre-production environment.
- Ensure Node and Next.js hosting support the current app runtime and route handlers.
- Confirm the host supports the current Next.js 16 app router and `proxy.ts` behavior.

## 2. Environment Variables

These values must be configured in the hosting provider instead of relying on local `.env.local`.

### Required now

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `NEXT_PUBLIC_SITE_URL`
- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `RESEND_API_KEY`
- `RESEND_FROM_EMAIL`

### Likely required later

- `STRIPE_SECRET_KEY`
- `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`
- `STRIPE_WEBHOOK_SECRET`

### Production notes

- `NEXT_PUBLIC_SITE_URL` must be the public HTTPS URL for the hosted app.
- Never point emails or RSVP links to `localhost` in production.
- All secrets should live in the host's secure environment settings, not in Git.

## 3. Supabase Production Readiness

- Ensure the production Supabase project exists.
- Link the local repo to the correct hosted Supabase project.
- Apply all migrations to production with `supabase db push`.
- Verify Row Level Security policies are active in production.
- Verify all required RPCs and triggers exist in production.
- Confirm `auth.users` profile creation trigger works in production.

### Auth settings

- Add the hosted site URL to Supabase Auth site settings.
- Add the hosted callback URL for auth flow.
- Add preview/staging callback URLs if we use them.
- Verify signup, login, logout, and callback redirect behavior on the hosted domain.

## 4. Email Delivery

- Replace the Resend sandbox sender with a verified sender/domain.
- Verify `RESEND_FROM_EMAIL` uses the production sender.
- Re-test invite send flow from the hosted site.
- Re-test inline invite card rendering in real inboxes.
- Re-test guest RSVP links from the hosted email.
- Confirm spam/authentication records are configured for the sending domain:
  - SPF
  - DKIM
  - DMARC

## 5. Public URLs And RSVP

- Confirm all RSVP links generated by the app use `NEXT_PUBLIC_SITE_URL`.
- Confirm public invite pages load correctly on the hosted domain.
- Confirm public card-image rendering works from hosted API routes.
- Test RSVP from a private/incognito browser session.
- Test RSVP from a mobile email client.

## 6. AI Integration

- Set production OpenAI credentials.
- Verify all AI routes succeed on the hosted site:
  - `/api/ai/generate-plan`
  - `/api/ai/generate-invite-copy`
  - `/api/ai/generate-shopping-list`
  - `/api/ai/revise-plan`
- Verify the fallback behavior is acceptable if OpenAI is unavailable.
- Re-check model routing against the Party Swami model design docs.
- Review production rate limiting and usage controls for AI endpoints.

## 7. Invite Template Hosting

- Confirm all invite pack assets are included in the production build.
- Confirm `public/invite-packs/*` are served correctly by the host.
- Confirm rendered card images match the in-app preview closely enough for production.
- Re-test long-copy handling in:
  - invitation generator preview
  - public RSVP page
  - email card rendering

## 8. Billing And Payments

These are not complete yet, but should stay on the hosting checklist.

- Replace placeholder Stripe webhook logic with real billing flow.
- Create Stripe products and prices.
- Configure hosted checkout/session flow.
- Configure Stripe webhook endpoint on the hosted site.
- Sync Stripe subscription state into app plan tiers.
- Re-test upgrade/downgrade/cancel behavior on production.

## 9. Observability And Safety

- Add production logging or error monitoring.
- Capture route-handler failures for:
  - auth
  - invite sending
  - AI generation
  - RSVP submission
- Add abuse protection or rate limiting for public and expensive endpoints.
- Review email sending limits and retry behavior.
- Review AI spend guardrails and usage enforcement in production.

## 10. QA Before Launch

### Host flow

- Sign up from the hosted site.
- Log in from the hosted site.
- Create an event.
- Select a template family during event creation.
- Open the event workspace.
- Edit guests, shopping items, tasks, and settings.
- Use AI plan generation.
- Rewrite invite message with AI.
- Save invite design.
- Send invite email.

### Guest flow

- Open invite email in a real inbox.
- Confirm the invite card image renders.
- Click RSVP link from email.
- Submit RSVP from the hosted public page.
- Confirm RSVP updates appear in the host dashboard/event workspace.

### Regression checks

- Dashboard event actions work.
- Back links between dashboard and event pages work.
- Public card rendering still works.
- Build passes:
  - `npm run lint`
  - `npm run typecheck`
  - `npm run build`
  - `npm run test:e2e`

## 11. Known Production Follow-Ups

- Replace any remaining `localhost` development assumptions.
- Add a staging deployment path if we want safer rollout testing.
- Improve fully automated signed-in production-like E2E coverage.
- Revisit invite email/card fidelity after hosted testing with real inboxes.

Use this section as a running ledger whenever we discover a new hosted-site requirement.

- Invite card images in email needed inline attachment handling because inboxes cannot fetch `localhost`.
